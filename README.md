# Doctrinal AAR Generator

Turns a wargame transcript into a structured After-Action Review with every claim cited to actual military doctrine, and every citation verified against a local corpus before the AAR is allowed to publish.

> SCSP Hackathon 2026 submission. Two demo inputs included; the system can ingest any wargame transcript that follows the Snow Globe `history` JSON shape.

## Problem

After-Action Reviews today take weeks, lose half their insight to author variance, and never get linked back to the doctrine they're supposed to test. GAO-23-105351 (March 2023) found that DoD wargames frequently produce findings that are difficult to act on because the analytical pipeline from observation to institutional change is broken — observations are recorded, but the path to a Lesson Identified, then to a Lesson Learned, then to a doctrine update is opaque and slow. The Special Competitive Studies Project's WarMatrix RFI explicitly calls for "*LLMs capable of facilitating real-time transcription and diarization of qualitative data (e.g., commander discussions) to enable rapid analysis during and after events*."

This project addresses the *interpretation* phase of wargaming, which Yuna Wong has identified as the underserved phase — the gap between a game ending and a usable AAR landing on the right desk in the right vocabulary.

## What this is

The system takes a wargame transcript (Snow Globe `history` JSON: ordered list of `{name, text}` entries by Blue Cell, Red Cell, Adjudicator, etc.) and produces a seven-section AAR that follows the format defined in the Department of the Army's "A Leader's Guide to After-Action Reviews" (December 2013), including the canonical SUSTAIN/IMPROVE dichotomy, the four-question structure, and a final Lessons Identified pass using NATO/JLLIS terminology.

Every doctrinal claim in the output carries an inline citation to a specific page of a specific doctrine document. Citations are not generated by the language model alone: each citation's `chunk_id` is round-tripped through the local doctrine index, and any unresolvable citation causes the section to be rejected. The user never sees a hallucinated source.

## Demo / how to run

Prerequisites: Python 3.13, an Anthropic API key.

```bash
# 1. Create venv and install requirements (uv if available is much faster)
uv venv --python 3.13 .venv
source .venv/bin/activate
uv pip install -r requirements.txt

# 2. Download the doctrine PDFs and demo-input source reports.
#    Some sources (.mil, .fas.org) are gated by Akamai/Cloudflare WAFs;
#    the script will skip those and continue. Drop any PDF you obtain
#    yourself into corpus/raw/ — the ingest pipeline picks it up automatically.
bash scripts/download_artifacts.sh

# 3. Index the corpus (PDF → 500-token chunks → embeddings → local Chroma index).
python -m src.ingest

# 4. Quick retrieval sanity check (no LLM needed).
python -m src.retrieval

# 5. Set your API key.
export ANTHROPIC_API_KEY="sk-ant-..."     # or put it in .env

# 6a. CLI: render one section.
python -m src.generate demo_inputs/csis_first_battle.json --section what_happened

# 6b. CLI: render the full AAR (seven sections + Lessons Identified).
python -m src.generate demo_inputs/csis_first_battle.json --full

# 6c. UI: launch the web server at http://127.0.0.1:8000/
python -m src.server
```

The web UI displays the AAR streaming in section by section. Citations are clickable — clicking one opens a modal showing the actual chunk of doctrine that was retrieved and used to support the claim. If a section fails the citation gate, the section is marked red in the UI and the offending chunk_ids are surfaced.

## Technical approach

PDF → page-aware chunks (preserving page numbers for citation accuracy) → MiniLM embeddings → ChromaDB. For each AAR section, a section-specific query retrieves the top-k chunks; those chunks plus the transcript are sent to `claude-opus-4-7` along with a prompt-engineered system message that enforces the citation format. Generated text is parsed for `{{cite:CHUNK_ID}}` tags, and every cited id is verified to exist in the corpus before the section is returned. Streaming is plumbed end-to-end (Anthropic stream → SSE → DOM updates), but verification happens after stream completion so a hallucinated citation aborts the section visibly.

## How this avoids the typical AI-project anti-patterns

The Ryseff *et al.* taxonomy (RAND **RR-A2680-1**, "The Root Causes of Failure for Artificial Intelligence Projects and How They Can Succeed") names the recurring failure modes for defense-AI projects: unclear problem framing, output that can't be traced or audited, scope sprawl, and deliverables that depend on infrastructure that isn't really there. This project is built against those failure modes deliberately:

- **Traceable model outputs.** Every AAR claim is linked to a specific page of a specific named manual. The link is not aspirational — it is parsed, validated against the index, and refused if it doesn't resolve. A judge or a doctrine cell can click through to verify.
- **Bounded scope.** The system does one thing: turn a wargame transcript into a doctrinally-cited AAR. It does not pretend to write doctrine, score wargames, or replace CALL/JLLIS. It produces better intake for those institutions.
- **Testable pipeline.** Retrieval can be inspected (`python -m src.retrieval` prints top hits with distances). Chunks are written to `corpus/chunks/*.json` for human review. Every citation gets round-tripped.
- **Real infrastructure, no demo magic.** The app is one Python process, one HTML page, one local vector index. There are no hidden cloud services, no synthesized "demo data" passed off as real, no models doing things the docs say they can't.

## Doctrinal grounding

The corpus is bounded and documented. Doctrine PDFs are placed in `corpus/raw/`; the ingest pipeline records the source manual, page range, and document type for every chunk it produces. The target corpus:

- **A Leader's Guide to After-Action Reviews** (US Army, December 2013) — the canonical AAR format reference.
- **JP 5-0 Joint Planning** — branches, sequels, decision points.
- **JP 3-0 Joint Operations** — joint functions, decisive/shaping operations.
- **FM 3-0 Operations** and **FM 5-0 Planning and Orders Production** — Army planning doctrine.
- **ATP 7-100.3 Chinese Tactics** — adversary doctrine for red-cell discussion.

The download script attempts each source automatically. Several official sources (`.mil`, `irp.fas.org`) are protected by Akamai/Cloudflare WAFs that block non-browser fetches; the script skips those gracefully. Add any PDFs you obtain manually to `corpus/raw/` and re-run `python -m src.ingest`.
